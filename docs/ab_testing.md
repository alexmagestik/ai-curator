# A/B тестирование промптов

Раздел **A/B тесты** (admin) сравнивает два системных промпта на одном и том же
вопросе при полностью идентичных остальных параметрах и собирает метрики качества
в рамках сессии.

## Зачем

- Проверить, улучшает ли новая формулировка системного промпта качество ответов.
- Оценить релевантность и склонность к галлюцинациям **вслепую**, без предвзятости
  к «текущему» или «новому» промпту.
- Получить сводные метрики (Relevance, Hallucinations, Relative Lift) по серии
  вопросов.

## Что фиксируется одинаковым

Меняется **только базовый системный промпт**. Всё остальное идентично для обоих
вариантов:

- один и тот же вопрос;
- один retrieval (выполняется один раз) → общий контекст и одинаковые документы;
- одинаковый уровень студента и фильтр по модулю;
- одна и та же модель и её параметры;
- одинаково добавляемые инструкции по уровню и few-shot примеры.

Код: [`app/rag/pipeline.py`](../app/rag/pipeline.py) → `RAGPipeline.compare_prompts()`
использует общий retrieval и вызывает генерацию дважды с разным
`base_system_prompt`. Сборка промпта — [`app/prompts/loader.py`](../app/prompts/loader.py)
→ `build_system_prompt(user_level=..., base_prompt=...)`.

## Интерфейс

1. **Вариант A** — текущий системный промпт (`system_prompt.txt`), поле только для
   чтения.
2. **Вариант B** — редактируемое поле, по умолчанию равно текущему промпту.
3. **Пользовательский промпт (вопрос)** — один на оба варианта.
4. Параметры: **уровень студента**, **фильтр по модулю**.
5. Кнопка **Сравнить ответы** отправляет оба запроса в модель.

### Слепая оценка

После сравнения ответы размещаются **случайно** слева/справа, вариант промпта
скрыт. Под каждым ответом:

| Чекбокс | Метрика | По умолчанию | Логика |
|---|---|---|---|
| **Relevance — ответ корректен** | Primary | включён | Ответ релевантен вопросу |
| **Hallucinations — модель галлюцинирует** | Guard | выключен | При включении Relevance снимается и блокируется; при выключении — возвращается прежнее значение |

Кнопка **Оценить** сохраняет результат текущего вопроса в сессию, сопоставляя
левый/правый выбор с реальными вариантами A/B, очищает поле вопроса и добавляет
строку в таблицу истории.

## Таблица истории

| Поле | Содержание |
|---|---|
| Вопрос | Текст вопроса (обрезка до 70 символов) |
| Модель A | Знак результата варианта A |
| Модель B | Знак результата варианта B |

Знаки результата:

| Знак | Значение |
|---|---|
| ✅ | Корректный (релевантный) ответ |
| ❌ | Нерелевантный ответ |
| 👻 | Галлюцинация (имеет приоритет над релевантностью) |

Знаки всегда привязаны к настоящему варианту A/B независимо от того, слева или
справа был показан ответ.

## Метрики сессии

Для каждого варианта (A и B) по всем оценённым вопросам:

- **Relevance** = количество корректных ответов / общее количество ответов.
- **Hallucinations** = количество ответов, где модель галлюцинировала (счётчик).

Сводная метрика:

- **Relative Lift (B vs A)** = `(Relevance B − Relevance A) / Relevance A × 100%`.
  Показывает прирост релевантности варианта B относительно A. При `Relevance A = 0`
  выводится «—» (защита от деления на ноль).

Также отображается счётчик «Оценено вопросов: N / 30».

## Сессия и сброс

- Результаты хранятся **в рамках текущей сессии браузера** (`st.session_state`),
  в базу данных не записываются.
- **Пропустить вопрос** — отменить текущее сравнение без записи.
- **Сбросить сессию** — очистить все результаты и начать заново.
- Лимит серии — **30 вопросов**; при достижении кнопка сравнения блокируется.

## Ограничения

- Оценка ручная и субъективная (эксперт отмечает релевантность/галлюцинации).
- Метрики не сохраняются между перезапусками приложения.
- Требуется `OPENAI_API_KEY` — оба ответа генерируются реальной моделью.
